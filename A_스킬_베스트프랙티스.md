# 스킬 작성 베스트 프랙티스

Claude가 발견하고 성공적으로 사용할 수 있는 효과적인 스킬을 작성하는 방법을 알아봅니다.

좋은 스킬은 간결하고, 잘 구조화되어 있으며, 실제 사용을 통해 검증된 것입니다. 이 가이드는 Claude가 효과적으로 발견하고 사용할 수 있는 스킬을 작성하는 데 도움이 되는 실용적인 작성 결정들을 제공합니다.

스킬의 작동 방식에 대한 개념적 배경은 [스킬 개요](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview)를 참고하세요.

## 핵심 원칙

### 간결함이 핵심

[컨텍스트 윈도우(context window)](https://platform.claude.com/docs/en/build-with-claude/context-windows)는 공공재입니다. 스킬은 Claude가 알아야 하는 다른 모든 것들과 컨텍스트 윈도우를 공유합니다:
- 시스템 프롬프트
- 대화 기록
- 다른 스킬들의 메타데이터
- 실제 요청 내용

스킬의 모든 토큰이 즉각적인 비용을 발생시키는 것은 아닙니다. 시작 시에는 모든 스킬의 메타데이터(이름과 설명)만 미리 로드됩니다. Claude는 스킬이 관련성이 있을 때만 SKILL.md를 읽고, 필요할 때만 추가 파일을 읽습니다. 그러나 SKILL.md에서의 간결함은 여전히 중요합니다. Claude가 로드하면 모든 토큰이 대화 기록 및 다른 컨텍스트와 경쟁하기 때문입니다.

**기본 가정: Claude는 이미 매우 똑똑합니다**

Claude가 이미 갖고 있지 않은 컨텍스트만 추가하세요. 각 정보에 의문을 제기하세요:
- "Claude가 이 설명을 정말 필요로 할까?"
- "Claude가 이미 알고 있다고 가정해도 될까?"
- "이 문단이 토큰 비용을 정당화할 만한가?"

**좋은 예: 간결한 버전** (약 50 토큰):

```
## PDF 텍스트 추출

텍스트 추출에는 pdfplumber를 사용합니다:

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
```

### 적절한 자유도 설정

작업의 취약성과 가변성에 맞춰 구체성 수준을 조정하세요.

**높은 자유도** (텍스트 기반 지시사항):

다음 경우에 사용:
- 여러 접근 방식이 유효할 때
- 결정이 컨텍스트에 따라 달라질 때
- 경험적 방법이 접근 방식을 안내할 때

예시:

```
## 코드 리뷰 프로세스

1. 코드 구조와 구성 분석
2. 잠재적 버그 또는 엣지 케이스 확인
3. 가독성과 유지보수성 개선 사항 제안
4. 프로젝트 컨벤션 준수 여부 확인
```

**중간 자유도** (매개변수가 있는 의사코드 또는 스크립트):

다음 경우에 사용:
- 선호하는 패턴이 존재할 때
- 어느 정도의 변형이 허용될 때
- 설정이 동작에 영향을 미칠 때

예시:

```
## 리포트 생성

이 템플릿을 사용하되 필요에 따라 커스터마이즈하세요:

```python
def generate_report(data, format="markdown", include_charts=True):
    # 데이터 처리
    # 지정된 형식으로 출력 생성
    # 선택적으로 시각화 포함
```
```

**낮은 자유도** (매개변수가 적거나 없는 구체적인 스크립트):

다음 경우에 사용:
- 작업이 취약하고 오류가 발생하기 쉬울 때
- 일관성이 중요할 때
- 특정 순서를 반드시 따라야 할 때

예시:

```
## 데이터베이스 마이그레이션

정확히 이 스크립트를 실행하세요:

```bash
python scripts/migrate.py --verify --backup
```

커맨드를 수정하거나 추가 플래그를 넣지 마세요.
```

**비유:** Claude를 경로를 탐험하는 로봇으로 생각해보세요:
- **양쪽에 절벽이 있는 좁은 다리:** 앞으로 나아가는 안전한 방법은 하나뿐입니다. 구체적인 가드레일과 정확한 지시사항을 제공하세요(낮은 자유도). 예: 정확한 순서로 실행되어야 하는 데이터베이스 마이그레이션.
- **위험 요소가 없는 탁 트인 들판:** 많은 경로가 성공으로 이어집니다. 일반적인 방향을 제시하고 Claude가 최선의 경로를 찾을 것을 믿으세요(높은 자유도). 예: 컨텍스트가 최선의 접근 방식을 결정하는 코드 리뷰.

### 사용 예정인 모든 모델로 테스트하기

스킬은 모델에 대한 추가 기능으로 작동하므로, 효과는 기반 모델에 따라 달라집니다. 사용할 모든 모델로 스킬을 테스트하세요.

**모델별 테스트 고려사항:**
- **Claude Haiku** (빠르고 경제적): 스킬이 충분한 안내를 제공하는가?
- **Claude Sonnet** (균형): 스킬이 명확하고 효율적인가?
- **Claude Opus** (강력한 추론): 스킬이 과도한 설명을 피하고 있는가?

Opus에서 완벽하게 작동하는 것이 Haiku에서는 더 많은 세부 정보가 필요할 수 있습니다. 여러 모델에서 스킬을 사용할 계획이라면 모든 모델에서 잘 작동하는 지시사항을 목표로 하세요.

## 스킬 구조

**YAML 프론트매터(frontmatter):** SKILL.md의 프론트매터에는 두 가지 필드가 필요합니다:

`name`:
- 최대 64자
- 소문자, 숫자, 하이픈만 포함 가능
- XML 태그 포함 불가
- 예약어 불가: "anthropic", "claude"

`description`:
- 비어 있으면 안 됨
- 최대 1024자
- XML 태그 포함 불가
- 스킬이 무엇을 하는지, 언제 사용하는지 설명해야 함

스킬 구조에 대한 전체 세부 정보는 [스킬 개요](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview#skill-structure)를 참고하세요.

### 네이밍 컨벤션

스킬을 참조하고 논의하기 쉽게 일관된 네이밍 패턴을 사용하세요. 스킬 이름에는 **동명사형(gerund form)** (동사 + -ing)을 사용하는 것을 고려해보세요. 이 형태가 스킬이 제공하는 활동이나 기능을 명확하게 설명하기 때문입니다.

`name` 필드는 소문자, 숫자, 하이픈만 사용해야 합니다.

**좋은 네이밍 예시 (동명사형):**
- `processing-pdfs`
- `analyzing-spreadsheets`
- `managing-databases`
- `testing-code`
- `writing-documentation`

**허용되는 대안:**
- 명사구: `pdf-processing`, `spreadsheet-analysis`
- 동작 지향: `process-pdfs`, `analyze-spreadsheets`

**피해야 할 것:**
- 모호한 이름: `helper`, `utils`, `tools`
- 지나치게 일반적: `documents`, `data`, `files`
- 예약어: `anthropic-helper`, `claude-tools`
- 스킬 컬렉션 내의 비일관적인 패턴

일관된 네이밍은 다음을 더 쉽게 만들어줍니다:
- 문서와 대화에서 스킬 참조하기
- 한눈에 스킬이 무엇을 하는지 이해하기
- 여러 스킬을 정리하고 검색하기
- 전문적이고 일관된 스킬 라이브러리 유지하기

### 효과적인 설명 작성

`description` 필드는 스킬 발견을 가능하게 하며, 스킬이 무엇을 하는지와 언제 사용하는지를 모두 포함해야 합니다.

**항상 3인칭으로 작성하세요.** 설명은 시스템 프롬프트에 삽입되며, 일관성 없는 관점은 발견 문제를 일으킬 수 있습니다.
- **좋음:** "Excel 파일을 처리하고 리포트를 생성합니다"
- **피할 것:** "저는 Excel 파일 처리를 도와드릴 수 있습니다"
- **피할 것:** "이것을 사용해 Excel 파일을 처리할 수 있습니다"

**구체적으로 작성하고 핵심 용어를 포함하세요.** 스킬이 무엇을 하는지, 언제 사용해야 하는지에 대한 구체적인 트리거/컨텍스트를 모두 포함하세요.

각 스킬에는 정확히 하나의 description 필드가 있습니다. 설명은 스킬 선택에 매우 중요합니다. Claude는 이를 사용해 100개 이상의 사용 가능한 스킬 중에서 올바른 스킬을 선택합니다. 설명은 Claude가 이 스킬을 선택해야 할 때를 알 수 있도록 충분한 세부 정보를 제공해야 하며, SKILL.md의 나머지 부분은 구현 세부 정보를 제공합니다.

효과적인 예시:

**PDF 처리 스킬:**

```
description: PDF 파일에서 텍스트와 표를 추출하고, 양식을 작성하며, 문서를 병합합니다. PDF 파일 작업 시 또는 사용자가 PDF, 양식, 문서 추출을 언급할 때 사용합니다.
```

**Excel 분석 스킬:**

```
description: Excel 스프레드시트를 분석하고, 피벗 테이블을 만들며, 차트를 생성합니다. Excel 파일, 스프레드시트, 표 형식 데이터, .xlsx 파일을 분석할 때 사용합니다.
```

**Git 커밋 도우미 스킬:**

```
description: git diff를 분석해 설명적인 커밋 메시지를 생성합니다. 사용자가 커밋 메시지 작성 또는 스테이징된 변경 사항 검토를 요청할 때 사용합니다.
```

다음과 같은 모호한 설명은 피하세요:

```
description: 문서를 도와줍니다
```

```
description: 데이터를 처리합니다
```

```
description: 파일로 무언가를 합니다
```

### 프로그레시브 디스클로저(progressive disclosure) 패턴

SKILL.md는 온보딩 가이드의 목차처럼 필요에 따라 Claude를 상세 자료로 안내하는 개요 역할을 합니다. 프로그레시브 디스클로저의 작동 방식에 대한 설명은 개요의 [스킬 작동 방식](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview#how-skills-work)을 참고하세요.

**실용적인 안내:**
- 최적 성능을 위해 SKILL.md 본문을 500줄 이하로 유지
- 이 한도에 근접하면 내용을 별도 파일로 분리
- 아래 패턴을 사용해 지시사항, 코드, 리소스를 효과적으로 구성

#### 시각적 개요: 단순한 것에서 복잡한 것으로

기본 스킬은 메타데이터와 지시사항이 담긴 SKILL.md 파일 하나로 시작합니다:

![YAML 프론트매터와 마크다운 본문을 보여주는 단순한 SKILL.md 파일](https://platform.claude.com/docs/images/agent-skills-simple-file.png)

스킬이 성장하면 Claude가 필요할 때만 로드하는 추가 내용을 번들로 묶을 수 있습니다:

![reference.md와 forms.md 같은 추가 참조 파일을 번들링하는 모습](https://platform.claude.com/docs/images/agent-skills-bundling-content.png)

완전한 스킬 디렉토리 구조는 다음과 같습니다:

```
pdf/
├── SKILL.md              # 주요 지시사항 (트리거 시 로드됨)
├── FORMS.md              # 양식 작성 가이드 (필요 시 로드됨)
├── reference.md          # API 참조 (필요 시 로드됨)
├── examples.md           # 사용 예시 (필요 시 로드됨)
└── scripts/
    ├── analyze_form.py   # 유틸리티 스크립트 (로드되지 않고 실행됨)
    ├── fill_form.py      # 양식 작성 스크립트
    └── validate.py       # 검증 스크립트
```

## 빠른 시작

pdfplumber로 텍스트 추출:
```python
import pdfplumber
with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```

## 고급 기능

**양식 작성**: 전체 가이드는 [FORMS.md](FORMS.md) 참고
**API 참조**: 모든 메서드는 [REFERENCE.md](REFERENCE.md) 참고
**예시**: 일반적인 패턴은 [EXAMPLES.md](EXAMPLES.md) 참고
```

Claude는 필요할 때만 FORMS.md, REFERENCE.md, 또는 EXAMPLES.md를 로드합니다.

#### 패턴 2: 도메인별 구성

여러 도메인을 가진 스킬의 경우, 관련 없는 컨텍스트가 로드되지 않도록 도메인별로 내용을 구성하세요. 사용자가 영업 지표에 대해 물어볼 때, Claude는 재무나 마케팅 데이터가 아닌 영업 관련 스키마만 읽으면 됩니다. 이렇게 하면 토큰 사용량이 낮게 유지되고 컨텍스트가 집중됩니다.

```
bigquery-skill/
├── SKILL.md (개요 및 탐색)
└── reference/
    ├── finance.md (매출, 청구 지표)
    ├── sales.md (기회, 파이프라인)
    ├── product.md (API 사용, 기능)
    └── marketing.md (캠페인, 기여)
```

SKILL.md:

```
# BigQuery 데이터 분석

## 사용 가능한 데이터셋

**재무**: 매출, ARR, 청구 → [reference/finance.md](reference/finance.md) 참고
**영업**: 기회, 파이프라인, 계정 → [reference/sales.md](reference/sales.md) 참고
**제품**: API 사용, 기능, 도입 → [reference/product.md](reference/product.md) 참고
**마케팅**: 캠페인, 기여, 이메일 → [reference/marketing.md](reference/marketing.md) 참고

## 빠른 검색

grep을 사용해 특정 지표 찾기:

```bash
grep -i "revenue" reference/finance.md
grep -i "pipeline" reference/sales.md
grep -i "api usage" reference/product.md
```
```

#### 패턴 3: 조건부 세부사항

기본 내용을 보여주고 고급 내용에 링크:

```
# DOCX 처리

### 깊게 중첩된 참조 피하기

Claude는 다른 참조된 파일에서 참조된 파일을 부분적으로 읽을 수 있습니다. 중첩된 참조를 만날 때, Claude는 전체 파일을 읽는 대신 내용을 미리보기 위해 `head -100` 같은 커맨드를 사용할 수 있어 불완전한 정보가 될 수 있습니다.

**SKILL.md에서 참조는 한 단계 깊이로 유지하세요.** 모든 참조 파일은 SKILL.md에서 직접 링크되어야 Claude가 필요할 때 완전한 파일을 읽을 수 있습니다.

**나쁜 예: 너무 깊은 중첩**:

```
# SKILL.md
[advanced.md](advanced.md) 참고...

# advanced.md
[details.md](details.md) 참고...

# details.md
실제 정보가 여기 있습니다...
```

**좋은 예: 한 단계 깊이**:

```
# SKILL.md

**기본 사용법**: [SKILL.md의 지시사항]
**고급 기능**: [advanced.md](advanced.md) 참고
**API 참조**: [reference.md](reference.md) 참고
**예시**: [examples.md](examples.md) 참고
```

### 긴 참조 파일에는 목차 구성하기

100줄 이상의 참조 파일에는 상단에 목차를 포함하세요. 이렇게 하면 Claude가 부분 읽기로 미리보기를 할 때도 사용 가능한 정보의 전체 범위를 파악할 수 있습니다.

**예시:**

```
# API 참조

## 목차
- 인증 및 설정
- 핵심 메서드 (생성, 읽기, 업데이트, 삭제)
- 고급 기능 (배치 작업, 웹훅)
- 오류 처리 패턴
- 코드 예시

## 인증 및 설정
...

## 핵심 메서드
...
```

이렇게 하면 Claude는 전체 파일을 읽거나 필요에 따라 특정 섹션으로 이동할 수 있습니다.

이 파일시스템 기반 아키텍처가 프로그레시브 디스클로저를 가능하게 하는 방법에 대한 자세한 내용은 아래 고급 섹션의 [런타임 환경](#런타임-환경) 섹션을 참고하세요.

## 워크플로우와 피드백 루프

### 복잡한 작업에는 워크플로우 사용하기

복잡한 작업을 명확하고 순차적인 단계로 분리하세요. 특히 복잡한 워크플로우의 경우, Claude가 응답에 복사하고 진행하면서 체크할 수 있는 체크리스트를 제공하세요.

**예시 1: 연구 종합 워크플로우** (코드가 없는 스킬용):

````
## 연구 종합 워크플로우

이 체크리스트를 복사하고 진행 상황을 추적하세요:

```
연구 진행 상황:
- [ ] 1단계: 모든 소스 문서 읽기
- [ ] 2단계: 핵심 테마 파악
- [ ] 3단계: 주장 교차 참조
- [ ] 4단계: 구조화된 요약 작성
- [ ] 5단계: 인용 검증
```

**1단계: 모든 소스 문서 읽기**

`sources/` 디렉토리의 각 문서를 검토하세요. 주요 주장과 뒷받침 증거를 기록하세요.

**2단계: 핵심 테마 파악**

소스 전반에서 패턴을 찾으세요. 반복적으로 나타나는 테마는 무엇인가요? 소스들이 어디서 동의하거나 이견을 보이나요?

**3단계: 주장 교차 참조**

각 주요 주장에 대해 소스 자료에 나타나는지 확인하세요. 각 포인트를 뒷받침하는 소스를 기록하세요.

**4단계: 구조화된 요약 작성**

테마별로 발견 사항을 정리하세요. 다음을 포함하세요:
- 주요 주장
- 소스의 뒷받침 증거
- 상충되는 관점 (있는 경우)

**5단계: 인용 검증**

모든 주장이 올바른 소스 문서를 참조하는지 확인하세요. 인용이 불완전하면 3단계로 돌아가세요.
````

이 예시는 코드가 필요 없는 분석 작업에 워크플로우를 어떻게 적용하는지 보여줍니다. 체크리스트 패턴은 복잡한 다단계 프로세스에 작동합니다.

**예시 2: PDF 양식 작성 워크플로우** (코드가 있는 스킬용):

````
## PDF 양식 작성 워크플로우

이 체크리스트를 복사하고 항목을 완료하면 체크하세요:

```
작업 진행 상황:
- [ ] 1단계: 양식 분석 (analyze_form.py 실행)
- [ ] 2단계: 필드 매핑 생성 (fields.json 편집)
- [ ] 3단계: 매핑 검증 (validate_fields.py 실행)
- [ ] 4단계: 양식 작성 (fill_form.py 실행)
- [ ] 5단계: 출력 검증 (verify_output.py 실행)
```

**1단계: 양식 분석**

실행: `python scripts/analyze_form.py input.pdf`

양식 필드와 위치를 추출해 `fields.json`에 저장합니다.

**2단계: 필드 매핑 생성**

`fields.json`을 편집해 각 필드에 값을 추가하세요.

**3단계: 매핑 검증**

실행: `python scripts/validate_fields.py fields.json`

계속하기 전에 검증 오류를 수정하세요.

**4단계: 양식 작성**

실행: `python scripts/fill_form.py input.pdf fields.json output.pdf`

**5단계: 출력 검증**

실행: `python scripts/verify_output.py output.pdf`

검증에 실패하면 2단계로 돌아가세요.
````

명확한 단계는 Claude가 중요한 검증을 건너뛰는 것을 방지합니다. 체크리스트는 Claude와 사용자 모두 다단계 워크플로우의 진행 상황을 추적하는 데 도움이 됩니다.

이 패턴은 출력 품질을 크게 향상시킵니다.

**예시 1: 스타일 가이드 준수** (코드가 없는 스킬용):

**예시 2: 문서 편집 프로세스** (코드가 있는 스킬용):

## 문서 편집 프로세스

1. `word/document.xml`에 편집 내용 적용
2. **즉시 검증**: `python ooxml/scripts/validate.py unpacked_dir/`

### 조건부 워크플로우 패턴

결정 지점을 통해 Claude를 안내하세요:

```
## 문서 수정 워크플로우

1. 수정 유형을 결정하세요:

   **새 내용 생성?** → 아래 "생성 워크플로우" 따르기
   **기존 내용 편집?** → 아래 "편집 워크플로우" 따르기

   - docx-js 라이브러리 사용

   - 기존 문서 압축 해제

   - 각 변경 후 검증
   - 완료 시 재압축

워크플로우가 많은 단계로 크거나 복잡해지는 경우, 별도 파일로 분리하고 현재 작업에 따라 적절한 파일을 읽도록 Claude에게 알려주세요.

## 평가와 반복

### 먼저 평가 구축하기

**광범위한 문서 작성 전에 평가를 먼저 만드세요.** 이렇게 하면 스킬이 가상의 문제가 아닌 실제 문제를 해결합니다.

**평가 주도 개발:**
1. **격차 파악:** 스킬 없이 대표적인 작업에서 Claude를 실행합니다. 구체적인 실패나 누락된 컨텍스트를 기록하세요
2. **평가 만들기:** 이러한 격차를 테스트하는 세 가지 시나리오를 구축하세요
3. **기준선 설정:** 스킬 없이 Claude의 성능을 측정하세요
4. **최소 지시사항 작성:** 격차를 해결하고 평가를 통과하기에 충분한 내용만 만드세요
5. **반복:** 평가를 실행하고, 기준선과 비교하고, 개선하세요

이 접근 방식은 실현되지 않을 수 있는 요건을 예상하는 것이 아니라 실제 문제를 해결하는 것을 보장합니다.

**평가 구조:**

```json
{
  "skills": ["pdf-processing"],
  "query": "이 PDF 파일에서 모든 텍스트를 추출하고 output.txt에 저장하세요",
  "files": ["test-files/document.pdf"],
  "expected_behavior": [
    "적절한 PDF 처리 라이브러리나 커맨드라인 도구를 사용해 PDF 파일을 성공적으로 읽습니다",
    "어떤 페이지도 누락하지 않고 문서의 모든 페이지에서 텍스트 내용을 추출합니다",
    "추출된 텍스트를 명확하고 읽기 쉬운 형식으로 output.txt라는 파일에 저장합니다"
  ]
}
```

이 예시는 간단한 테스트 기준이 있는 데이터 주도 평가를 보여줍니다. 현재는 이러한 평가를 실행하는 내장 방법이 없습니다. 사용자가 자체 평가 시스템을 만들 수 있습니다. 평가는 스킬 효과성을 측정하는 진실의 원천입니다.

### Claude와 함께 스킬을 반복적으로 개발하기

가장 효과적인 스킬 개발 프로세스는 Claude 자체를 포함합니다. Claude의 한 인스턴스("Claude A")와 함께 다른 인스턴스("Claude B")가 사용하는 스킬을 만드세요. Claude A는 지시사항 설계 및 개선을 도와주고, Claude B는 실제 작업에서 테스트합니다. Claude 모델이 효과적인 에이전트 지시사항을 작성하는 방법과 에이전트가 필요로 하는 정보를 모두 이해하기 때문에 이 방법이 효과적입니다.

**새 스킬 만들기:**

1. **스킬 없이 작업 완료:** 일반 프롬프팅을 사용해 Claude A와 함께 문제를 해결합니다. 작업하면서 자연스럽게 컨텍스트를 제공하고, 선호도를 설명하고, 절차적 지식을 공유하게 됩니다. 반복적으로 제공하는 정보가 무엇인지 확인하세요.

2. **재사용 가능한 패턴 파악:** 작업 완료 후, 미래의 유사한 작업에 유용할 컨텍스트를 파악하세요.

   **예시:** BigQuery 분석을 진행했다면 테이블 이름, 필드 정의, 필터링 규칙("항상 테스트 계정 제외" 같은), 일반적인 쿼리 패턴을 제공했을 수 있습니다.

3. **Claude A에게 스킬 만들기 요청:** "방금 우리가 사용한 이 BigQuery 분석 패턴을 캡처하는 스킬을 만들어주세요. 테이블 스키마, 네이밍 컨벤션, 테스트 계정 필터링 규칙을 포함하세요."

   Claude 모델은 스킬 형식과 구조를 본래적으로 이해합니다. Claude가 스킬 만드는 것을 도와주도록 특별한 시스템 프롬프트나 "스킬 작성" 스킬이 필요하지 않습니다. 단순히 Claude에게 스킬을 만들어 달라고 요청하면 적절한 프론트매터와 본문 내용이 있는 올바른 구조의 SKILL.md 내용을 생성합니다.

4. **간결함 검토:** Claude A가 불필요한 설명을 추가하지 않았는지 확인하세요. "win rate가 무엇인지에 대한 설명을 제거해주세요 — Claude는 이미 알고 있습니다."라고 물어보세요.

5. **정보 아키텍처 개선:** Claude A에게 내용을 더 효과적으로 구성하도록 요청하세요. 예: "테이블 스키마가 별도 참조 파일에 있도록 구성해주세요. 나중에 더 많은 테이블을 추가할 수 있습니다."

6. **유사 작업에서 테스트:** 관련 사용 사례에서 Claude B(스킬이 로드된 새 인스턴스)와 함께 스킬을 사용하세요. Claude B가 올바른 정보를 찾고, 규칙을 올바르게 적용하며, 작업을 성공적으로 처리하는지 관찰하세요.

7. **관찰에 기반해 반복:** Claude B가 어려움을 겪거나 무언가를 놓친다면, 구체적인 내용으로 Claude A로 돌아오세요: "이 스킬을 사용할 때 Claude가 Q4 날짜별 필터링을 잊었습니다. 날짜 필터링 패턴에 대한 섹션을 추가해야 할까요?"

**기존 스킬 반복 개선:**

같은 계층적 패턴이 스킬을 개선할 때도 계속됩니다. 다음을 번갈아 가며 작업합니다:
- **Claude A와 작업** (스킬 개선을 도와주는 전문가)
- **Claude B로 테스트** (실제 작업을 수행하기 위해 스킬을 사용하는 에이전트)
- **Claude B의 행동 관찰** 및 인사이트를 Claude A로 가져오기

1. **실제 워크플로우에서 스킬 사용:** Claude B(스킬이 로드된)에게 테스트 시나리오가 아닌 실제 작업 부여

2. **Claude B의 행동 관찰:** 어디서 어려움을 겪는지, 성공하는지, 예상치 못한 선택을 하는지 기록

   **관찰 예시:** "Claude B에게 지역 영업 리포트를 요청했을 때, 스킬에 이 규칙이 언급되어 있음에도 불구하고 테스트 계정 필터링을 잊었습니다."

3. **개선을 위해 Claude A로 돌아오기:** 현재 SKILL.md를 공유하고 관찰 내용을 설명하세요. "Claude B에게 지역 리포트를 요청했을 때 테스트 계정 필터링을 잊었다는 것을 발견했습니다. 스킬에 필터링이 언급되어 있지만 충분히 눈에 띄지 않을 수 있습니다?"라고 물어보세요.

4. **Claude A의 제안 검토:** Claude A는 규칙을 더 눈에 띄게 재구성하거나, "always filter" 대신 "MUST filter" 같은 더 강한 언어를 사용하거나, 워크플로우 섹션을 재구성하는 것을 제안할 수 있습니다.

5. **변경 사항 적용 및 테스트:** Claude A의 개선 사항으로 스킬을 업데이트한 다음 유사한 요청에서 Claude B로 다시 테스트하세요

6. **사용에 따라 반복:** 새로운 시나리오를 만날 때마다 관찰-개선-테스트 주기를 계속하세요. 각 반복은 가정이 아닌 실제 에이전트 행동을 기반으로 스킬을 개선합니다.

**팀 피드백 수집:**
1. 동료들과 스킬을 공유하고 사용 패턴 관찰
2. 물어보세요: 스킬이 예상대로 활성화되는가? 지시사항이 명확한가? 무엇이 빠져 있는가?
3. 자신의 사용 패턴에서의 맹점을 해결하기 위해 피드백 반영

**이 접근 방식이 효과적인 이유:** Claude A는 에이전트 요구 사항을 이해하고, 사용자는 도메인 전문성을 제공하며, Claude B는 실제 사용을 통해 격차를 드러내고, 반복적인 개선은 가정이 아닌 관찰된 행동을 기반으로 스킬을 향상시킵니다.

### Claude가 스킬을 탐색하는 방법 관찰하기

스킬을 반복 개선하면서 Claude가 실제로 스킬을 어떻게 사용하는지 주의 깊게 살펴보세요. 다음을 확인하세요:
- **예상치 못한 탐색 경로:** Claude가 예상하지 못한 순서로 파일을 읽는가? 이는 구조가 생각만큼 직관적이지 않다는 것을 나타낼 수 있습니다
- **놓친 연결:** Claude가 중요한 파일에 대한 참조를 따르지 못하는가? 링크가 더 명확하거나 눈에 띄어야 할 수 있습니다
- **특정 섹션에 과도한 의존:** Claude가 같은 파일을 반복적으로 읽는다면, 해당 내용이 주요 SKILL.md에 있어야 할 수 있습니다
- **무시된 내용:** Claude가 번들된 파일에 전혀 접근하지 않는다면, 불필요하거나 주요 지시사항에서 신호가 불충분한 것일 수 있습니다

가정이 아닌 이러한 관찰을 기반으로 반복 개선하세요. 스킬 메타데이터의 `name`과 `description`은 특히 중요합니다. Claude는 현재 작업에 응답해 스킬을 트리거할지 여부를 결정할 때 이것을 사용합니다. 스킬이 무엇을 하는지, 언제 사용해야 하는지를 명확하게 설명하는지 확인하세요.

## 고급: 실행 가능한 코드가 포함된 스킬

아래 섹션은 실행 가능한 스크립트를 포함하는 스킬에 집중합니다. 스킬이 마크다운 지시사항만 사용한다면 [효과적인 스킬을 위한 체크리스트](#효과적인-스킬을-위한-체크리스트)로 건너뛰세요.

### 해결하기, 떠넘기지 않기

스킬을 위한 스크립트를 작성할 때, Claude에게 떠넘기는 대신 오류 조건을 처리하세요.

**좋은 예: 명시적으로 오류 처리:**

```python
def process_file(path):
    """파일을 처리하고, 존재하지 않으면 생성합니다."""
    try:
        with open(path) as f:
            return f.read()
    except FileNotFoundError:
        # 실패하는 대신 기본 내용으로 파일 생성
        print(f"파일 {path}를 찾을 수 없어 기본값으로 생성합니다")
        with open(path, "w") as f:
            f.write("")
        return ""
    except PermissionError:
        # 실패하는 대신 대안 제공
        print(f"{path}에 접근할 수 없어 기본값 사용")
        return ""
```

**나쁜 예: Claude에게 떠넘기기:**

```python
def process_file(path):
    # 그냥 실패하고 Claude가 알아서 하도록 내버려두기
    return open(path).read()
```

설정 매개변수도 "부두 상수"(Ousterhout의 법칙)를 피하기 위해 정당화되고 문서화되어야 합니다. 올바른 값을 모른다면 Claude가 어떻게 결정할까요?

**좋은 예: 자기 문서화:**

```python
# HTTP 요청은 일반적으로 30초 이내에 완료됩니다
# 더 긴 타임아웃은 느린 연결을 고려합니다
REQUEST_TIMEOUT = 30

# 세 번의 재시도는 신뢰성과 속도의 균형을 맞춥니다
# 대부분의 일시적 실패는 두 번째 재시도 때 해결됩니다
MAX_RETRIES = 3
```

**나쁜 예: 매직 넘버:**

```python
TIMEOUT = 47  # 왜 47?
RETRIES = 5  # 왜 5?
```

### 유틸리티 스크립트 제공하기

Claude가 스크립트를 작성할 수 있다고 해도, 미리 만들어진 스크립트는 장점이 있습니다:

**유틸리티 스크립트의 장점:**
- 생성된 코드보다 더 신뢰할 수 있습니다
- 토큰을 절약합니다 (컨텍스트에 코드를 포함할 필요 없음)
- 시간을 절약합니다 (코드 생성 필요 없음)
- 사용 전반에 걸쳐 일관성을 보장합니다

![지시사항 파일과 함께 실행 가능한 스크립트를 번들링하는 모습](https://platform.claude.com/docs/images/agent-skills-executable-scripts.png)

위 다이어그램은 실행 가능한 스크립트가 지시사항 파일과 어떻게 함께 작동하는지 보여줍니다. 지시사항 파일(forms.md)이 스크립트를 참조하며, Claude는 내용을 컨텍스트에 로드하지 않고 실행할 수 있습니다.

**중요한 구분:** Claude가 다음 중 무엇을 해야 하는지 지시사항에 명확하게 하세요:
- **스크립트 실행** (가장 일반적): "필드 추출하기 위해 `analyze_form.py` 실행"
- **참조로 읽기** (복잡한 로직의 경우): "필드 추출 알고리즘은 `analyze_form.py` 참고"

대부분의 유틸리티 스크립트에서는 더 신뢰할 수 있고 효율적이기 때문에 실행이 선호됩니다. 스크립트 실행의 작동 방식에 대한 자세한 내용은 아래 [런타임 환경](#런타임-환경) 섹션을 참고하세요.

**예시:**

````
## 유틸리티 스크립트

**analyze_form.py**: PDF에서 모든 양식 필드 추출

```bash
python scripts/analyze_form.py input.pdf > fields.json
```

출력 형식:
```json
{
  "field_name": {"type": "text", "x": 100, "y": 200},
  "signature": {"type": "sig", "x": 150, "y": 500}
}
```

**validate_boxes.py**: 겹치는 경계 상자 확인

```bash
python scripts/validate_boxes.py fields.json
# 반환: "OK" 또는 충돌 목록
```

**fill_form.py**: PDF에 필드 값 적용

```bash
python scripts/fill_form.py input.pdf fields.json output.pdf
```
````

### 시각적 분석 활용하기

입력을 이미지로 렌더링할 수 있는 경우, Claude가 분석하도록 하세요:

```
## 양식 레이아웃 분석

1. PDF를 이미지로 변환:
   ```bash
   python scripts/pdf_to_images.py form.pdf
   ```

2. 각 페이지 이미지를 분석해 양식 필드 파악
3. Claude가 필드 위치와 유형을 시각적으로 확인할 수 있습니다
```

이 예시에서는 `pdf_to_images.py` 스크립트를 직접 작성해야 합니다.

Claude의 비전 기능은 레이아웃과 구조를 이해하는 데 도움이 됩니다.

### 검증 가능한 중간 출력 만들기

Claude가 복잡하고 개방형 작업을 수행할 때 실수를 할 수 있습니다. "계획-검증-실행" 패턴은 Claude가 먼저 구조화된 형식으로 계획을 만들고, 실행하기 전에 스크립트로 계획을 검증함으로써 오류를 조기에 포착합니다.

**예시:** 스프레드시트를 기반으로 PDF의 50개 양식 필드를 업데이트하도록 Claude에게 요청한다고 상상해보세요. 검증 없이, Claude는 존재하지 않는 필드를 참조하거나, 충돌하는 값을 생성하거나, 필수 필드를 놓치거나, 업데이트를 잘못 적용할 수 있습니다.

**해결책:** 위에 표시된 워크플로우 패턴(PDF 양식 작성)을 사용하되, 변경 사항을 적용하기 전에 검증하는 중간 `changes.json` 파일을 추가하세요. 워크플로우는 다음과 같이 됩니다: 분석 → **계획 파일 생성** → **계획 검증** → 실행 → 검증.

**이 패턴이 효과적인 이유:**
- **오류 조기 포착:** 변경 사항이 적용되기 전에 검증이 문제를 찾습니다
- **기계 검증 가능:** 스크립트가 객관적인 검증을 제공합니다
- **역방향 계획:** Claude가 원본을 건드리지 않고 계획을 반복할 수 있습니다
- **명확한 디버깅:** 오류 메시지가 특정 문제를 가리킵니다

**사용 시기:** 배치 작업, 파괴적 변경, 복잡한 검증 규칙, 고위험 작업.

**구현 팁:** "Field 'signature_date' not found. Available fields: customer_name, order_total, signature_date_signed"와 같은 구체적인 오류 메시지로 검증 스크립트를 상세하게 만들어 Claude가 문제를 수정할 수 있도록 하세요.

### 패키지 의존성

스킬은 플랫폼별 제한이 있는 코드 실행 환경에서 실행됩니다:
- **claude.ai:** npm과 PyPI에서 패키지를 설치하고 GitHub 저장소에서 가져올 수 있습니다
- **Claude API:** 네트워크 접근이 없고 런타임 패키지 설치가 없습니다

SKILL.md에 필요한 패키지를 나열하고 [코드 실행 도구 문서](https://platform.claude.com/docs/en/agents-and-tools/tool-use/code-execution-tool)에서 사용 가능한지 확인하세요.

### 런타임 환경

스킬은 파일시스템 접근, bash 커맨드, 코드 실행 기능을 갖춘 코드 실행 환경에서 실행됩니다. 이 아키텍처의 개념적 설명은 개요의 [스킬 아키텍처](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview#the-skills-architecture)를 참고하세요.

**작성 방법에 미치는 영향:**

**Claude의 스킬 접근 방식:**
1. **메타데이터 미리 로드:** 시작 시, 모든 스킬의 YAML 프론트매터에서 이름과 설명이 시스템 프롬프트에 로드됩니다
2. **온디맨드 파일 읽기:** Claude는 필요할 때 파일시스템에서 SKILL.md와 다른 파일에 접근하기 위해 bash Read 도구를 사용합니다
3. **효율적인 스크립트 실행:** 유틸리티 스크립트는 전체 내용을 컨텍스트에 로드하지 않고 bash를 통해 실행될 수 있습니다. 스크립트의 출력만 토큰을 소비합니다
4. **대용량 파일에 대한 컨텍스트 패널티 없음:** 참조 파일, 데이터, 문서는 실제로 읽힐 때까지 컨텍스트 토큰을 소비하지 않습니다

- **파일 경로 중요:** Claude는 파일시스템처럼 스킬 디렉토리를 탐색합니다. 백슬래시가 아닌 슬래시를 사용하세요 (`reference/guide.md`)
- **파일을 설명적으로 이름 짓기:** 내용을 나타내는 이름 사용: `form_validation_rules.md`, `doc2.md` 아님
- **발견을 위해 구성:** 도메인이나 기능별로 디렉토리 구조화
  - 좋음: `reference/finance.md`, `reference/sales.md`
  - 나쁨: `docs/file1.md`, `docs/file2.md`
- **포괄적인 리소스 번들링:** 완전한 API 문서, 광범위한 예시, 대용량 데이터셋 포함 — 접근되기 전까지 컨텍스트 패널티 없음
- **결정론적 작업에는 스크립트 선호:** Claude에게 검증 코드를 생성하도록 요청하는 것보다 `validate_form.py` 작성
- **실행 의도 명확하게:** 
  - "필드 추출을 위해 `analyze_form.py` 실행" (실행)
  - "추출 알고리즘은 `analyze_form.py` 참고" (참조로 읽기)
- **파일 접근 패턴 테스트:** 실제 요청으로 테스트해 Claude가 디렉토리 구조를 탐색할 수 있는지 확인

**예시:**

```
bigquery-skill/
├── SKILL.md (개요, 참조 파일을 가리킴)
└── reference/
    ├── finance.md (매출 지표)
    ├── sales.md (파이프라인 데이터)
    └── product.md (사용 분석)
```

사용자가 매출에 대해 물어볼 때, Claude는 SKILL.md를 읽고, `reference/finance.md`에 대한 참조를 확인하고, bash를 호출해 해당 파일만 읽습니다. sales.md와 product.md 파일은 파일시스템에 남아 있어 필요할 때까지 컨텍스트 토큰을 전혀 소비하지 않습니다. 이 파일시스템 기반 모델이 프로그레시브 디스클로저를 가능하게 합니다. Claude는 각 작업에 필요한 것만 정확하게 탐색하고 선택적으로 로드할 수 있습니다.

기술 아키텍처에 대한 자세한 내용은 스킬 개요의 [스킬 작동 방식](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview#how-skills-work)을 참고하세요.

### MCP 도구 참조

스킬이 MCP(Model Context Protocol) 도구를 사용하는 경우, "도구를 찾을 수 없음" 오류를 피하기 위해 항상 완전히 한정된 도구 이름을 사용하세요.

**형식:** `ServerName:tool_name`

**예시:**

```
테이블 스키마를 검색하려면 BigQuery:bigquery_schema 도구를 사용하세요.
이슈를 생성하려면 GitHub:create_issue 도구를 사용하세요.
```

여기서:
- `BigQuery`와 `GitHub`는 MCP 서버 이름입니다
- `bigquery_schema`와 `create_issue`는 해당 서버 내의 도구 이름입니다

서버 접두사가 없으면, 특히 여러 MCP 서버가 사용 가능한 경우 Claude가 도구를 찾지 못할 수 있습니다.

### 도구가 설치되어 있다고 가정하지 않기

패키지가 사용 가능하다고 가정하지 마세요:

```
**나쁜 예: 설치를 가정함**:
"pdf 라이브러리를 사용해 파일을 처리하세요."

**좋은 예: 의존성에 대해 명확함**:
"필요한 패키지 설치: `pip install pypdf`

그런 다음 사용:
```python
from pypdf import PdfReader
reader = PdfReader("file.pdf")
```"
```

## 기술 참고사항

### YAML 프론트매터 요구사항

SKILL.md 프론트매터에는 특정 검증 규칙이 있는 `name`과 `description` 필드가 필요합니다:
- `name`: 최대 64자, 소문자/숫자/하이픈만, XML 태그 없음, 예약어 없음
- `description`: 최대 1024자, 비어 있으면 안 됨, XML 태그 없음

전체 구조 세부 사항은 [스킬 개요](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview#skill-structure)를 참고하세요.

### 토큰 예산

최적 성능을 위해 SKILL.md 본문을 500줄 이하로 유지하세요. 내용이 이를 초과하면 앞서 설명한 프로그레시브 디스클로저 패턴을 사용해 별도 파일로 분리하세요. 아키텍처 세부 사항은 [스킬 개요](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview#how-skills-work)를 참고하세요.

## 효과적인 스킬을 위한 체크리스트

스킬을 공유하기 전에 다음을 확인하세요:

### 핵심 품질
- [ ] 설명이 구체적이고 핵심 용어를 포함합니다
- [ ] 설명에 스킬이 무엇을 하는지와 언제 사용하는지가 모두 포함되어 있습니다
- [ ] SKILL.md 본문이 500줄 이하입니다
- [ ] 추가 세부사항이 별도 파일에 있습니다 (필요한 경우)
- [ ] 시간에 민감한 정보가 없습니다 (또는 "이전 패턴" 섹션에 있습니다)
- [ ] 전체적으로 일관된 용어를 사용합니다
- [ ] 예시가 추상적이지 않고 구체적입니다
- [ ] 파일 참조가 한 단계 깊이입니다
- [ ] 프로그레시브 디스클로저가 적절히 사용됩니다
- [ ] 워크플로우에 명확한 단계가 있습니다

### 코드와 스크립트
- [ ] 스크립트가 Claude에게 떠넘기지 않고 문제를 해결합니다
- [ ] 오류 처리가 명시적이고 도움이 됩니다
- [ ] "부두 상수"가 없습니다 (모든 값이 정당화됨)
- [ ] 필요한 패키지가 지시사항에 나열되고 사용 가능한지 확인되었습니다
- [ ] 스크립트에 명확한 문서가 있습니다
- [ ] Windows 스타일 경로가 없습니다 (모두 슬래시 사용)
- [ ] 중요한 작업에 검증/확인 단계가 포함됩니다
- [ ] 품질 중요 작업에 피드백 루프가 포함됩니다

### 테스트
- [ ] 최소 세 가지 평가가 만들어졌습니다
- [ ] Haiku, Sonnet, Opus로 테스트되었습니다
- [ ] 실제 사용 시나리오로 테스트되었습니다
- [ ] 팀 피드백이 반영되었습니다 (해당하는 경우)
